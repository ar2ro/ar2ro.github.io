Elements

<meta>

Placed inside the head, it's used to add metadata (data that describes data) to a document.

The <meta charset="utf-8"> specifies the character set that the document is permited to use (utf-8 includes pretty much any character from any human language) and it should always be used.

There are many other variations of the <meta> element available and they commonly include name and content attributes:

  • name specifies the type of meta element it is, what type of information it contains.
  • content specifies the actual meta content.
Example: <meta name="author" content="Chris Mills">

There are other types of metadata which are propietary creations like facebook's Open Graph Data or Twitter's cards. These are used to provide these websites with specific pieces of information they can use

Icons

You can add references to custom icons in your metadata and these will be displayed in certain contexts, example: providing a nice high resolution icon to use when the website is saved to an iPad's home screen.

Icons must be saved in the same directory as the site's index page in .ico format and is added to the head with the link element: <link rel="shortcut icon" href="favicon.ico" type="image/x-icon">

<b>, <i>, and <u>

These came about so people could write bold, italics, or underlined text in an era when CSS was still supported poorly or not at all.

HTML5 redefined <b>, <i>, and <u> with new, somewhat confusing, semantic roles.

<address>

It simply wraps around your contact details, for example: {% highlight html %}

Chris Mills, Manchester, The Grim North, UK

{% endhighlight %}

One thing to remember however is that the <address> element is meant for marking up the contact details of the person who wrote the HTML document, not any address. So the above would only be ok if Chris had written the document the markup appears on. Note that something like this would also be ok: {% highlight html %}

Page written by Chris Mills.

{% endhighlight %}

  • <pre>: For retaining whitespace (generally code blocks)
  • <var>: For specifically marking up variable names.
  • <kbd>: For marking up keyboard (and other types of) input entered into the computer.
  • <samp>: For marking up the output of a computer program.

<time>

It's used for marking up times and dates in a machine-readable format. For example: <time datetime="2016-01-20">20 January 2016</time>

There are many different ways that humans write down dates. But these different forms cannot be easily recognised by the computers — what if you wanted to automatically grab the dates of all events in a page and insert them into a calendar? The element allows you to attach an unambiguous, machine-readable time/date for this purpose. for example:

Standard simple date
<time datetime="2016-01-20">20 January 2016</time>
Just year and month
<time datetime="2016-01">January 2016</time>
Just month and day
<time datetime="01-20">20 January</time>
Just time, hours and minutes
<time datetime="19:30">19:30</time>
You can do seconds and milliseconds too!
<time datetime="19:30:01.856">19:30:01.856</time>
Date and time
<time datetime="2016-01-20T19:30">7.30pm, 20 January 2016</time>
Date and time with timezone offset
<time datetime="2016-01-20T19:30+01:00">7.30pm, 20 January 2016 is 8.30pm in France</time>
Calling out a specific week number
<time datetime="2016-W04">The fourth week of 2016</time>

lang

Helps define the language of an element: the language that non-editable elements are written in, or the language that the editable elements should be written in by the user.

A language tag is made of hyphen-separated language subtags, where each subtag indicates a certain property of the language. The 3 most common subtags are:

Language subtag
Required. A 2-or-3-character code that defines the basic language, typically written in all lowercase. For example, the language code for English is en, and the code for Badeshi is bdz.
Script subtag
Optional. This subtag defines the writing system used for the language, and is always 4 characters long, with the first letter capitalized. For example, French-in-Braille is fr-Brail and ja-Kana is Japanese written with the Katakana alphabet. If the language is written in a highly typical way, like English in the Latin alphabet, there is no need to use this subtag.
Region subtag
Optional. This subtag defines a dialect of the base language from a particular location, and is either 2 letters in ALLCAPS matching a country code, or 3 numbers matching a non-country area. For example, es-ES is for Spanish as spoken in Spain, and es-013 is Spanish as spoken in Central America. “International Spanish” would just be es.

To find the correct subtags visit:

  • https://r12a.github.io/app-subtags/
  • http://www.iana.org/assignments/language-subtag-registry/language-subtag-registry

For the CSS pseudo-class :lang, two invalid language names are different if their names are different. So while :lang(es) matches both lang="es-ES" and lang="es-419", :lang(xyzzy) would not match lang="xyzzy-Zorp!".

You should keep the tag as short as possible. For instance, use ja for Japanese and not ja-JP, unless there is a particular reason that you need to say that this is Japanese as spoken in Japan, rather than elsewhere.

Tip: You should always set the language of your page on the <html> element.

Detailed spec: w3 language tags

title

Contains text representing advisory information, related to the element it belongs to. Some typical uses:

  • Labeling <iframe> elements.
  • Providing an inline programmatically associated label for an <input> element.
  • Labeling controls in data tables.

If this attribute is omitted, it means that the title of the nearest ancestor of this element is still relevant (and could be used as the tooltip for that element). If this attribute is set to the empty string, it explicitly means its nearest ancestor's title is not relevant (and shouldn't be used in the tooltip for this element).

Additional semantics are attached to the title attributes of the following elements:

<link>
The title attribute is used to define a preferred or an alternate stylesheet.
Any stylesheet in a document falls into one of the following categories:
  • Persistent (no rel="alternate", no title=""): always applies to the document.
  • Preferred (no rel="alternate", with title="..." specified): applied by default, but disabled if an alternate stylesheet is selected. There can only be one preferred stylesheet, so providing stylesheets with different title attributes will cause some of them to be ignored.
  • Alternate (rel="stylesheet alternate", title="..." must be specified): disabled by default, can be selected.
<abbr>
The title attribute must contain a full human readable description or expansion of the abbreviation
Tip: The title attribute is not required in abbreviations, you can use <abbr> to semantically note an abbreviation without providing an expansion or description.
<input>
The title attribute is used in tandem with the pattern attribute, which is a regular expression that the control's value is checked against. The title attribute describes the pattern to help the user.

It's a good idea to add titles to links to provide supplementary useful information about the link, such as what kind of information the page contains, or things to be aware of.

Tip: Links are revealed on mouse hover which are not shown while using keyboard controls so if the title info is important to the usability of the page you should put it in the regular text

download

When you are linking to a resource that is to be downloaded rather than opened in the browser, you can use the download attribute to provide a default save filename. Example: {% highlight html %} Download Latest Firefox for Windows (64-bit) (English, US) {% endhighlight %}

mailto <href> value

In its most basic and commonly used form, a mailto: link simply indicates the email address of the intended recipient. For example: <a href="mailto:nowhere@mozilla.org">Send email to nowhere</a>

In fact, the email address is even optional. If you leave it out (that is, your href is simply "mailto:"), a new outgoing email window will be opened by the user's mail client that has no destination address specified yet.

In addition to the email address, you can provide other information. In fact, any standard mail header fields can be added to the mailto URL you provide. The most commonly used of these are "subject", "cc", and "body" (which is not a true header field, but allows you to specify a short content message for the new email). Each field and its value is specified as a query term.

Here's an example that includes a cc, bcc, subject and body: {% highlight html %} Send mail with cc, bcc, subject and body {% endhighlight %}

The values of each field must be URL-encoded, that is with non-printing characters (invisible characters like tabs, carriage returns, and page breaks) and spaces percent-escaped (Percent encoding). Also note the use of the question mark (?) to separate the main URL from the field values, and ampersands (&) to separate each field in the mailto: URL. This is standard URL query notation. Read The GET method (The GET method) to understand what URL query notation is more comonly used for. You can specify multiple addresses.
