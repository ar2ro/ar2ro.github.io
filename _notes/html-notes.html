Elements

<meta>

Placed inside the head, it's used to add metadata (data that describes data) to a document.

The <meta charset="utf-8"> specifies the character set that the document is permited to use (utf-8 includes pretty much any character from any human language) and it should always be used.

There are many other variations of the <meta> element available and they commonly include name and content attributes:

  • name specifies the type of meta element it is, what type of information it contains.
  • content specifies the actual meta content.

Example: <meta name="author" content="Chris Mills">

There are other types of metadata which are propietary creations like facebook's Open Graph Data or Twitter's cards. These are used to provide these websites with specific pieces of information they can use

<b>, <i>, and <u>

These came about so people could write bold, italics, or underlined text in an era when CSS was still supported poorly or not at all.

HTML5 redefined <b>, <i>, and <u> with new, somewhat confusing, semantic roles.

<time>

It's used for marking up times and dates in a machine-readable format. For example: <time datetime="2016-01-20">20 January 2016</time>

There are many different ways that humans write down dates. But these different forms cannot be easily recognised by the computers — what if you wanted to automatically grab the dates of all events in a page and insert them into a calendar? The <time> element allows you to attach an unambiguous, machine-readable time/date for this purpose. for example:

Standard simple date
<time datetime="2016-01-20">20 January 2016</time>
Just year and month
<time datetime="2016-01">January 2016</time>
Just month and day
<time datetime="01-20">20 January</time>
Just time, hours and minutes
<time datetime="19:30">19:30</time>
You can do seconds and milliseconds too!
<time datetime="19:30:01.856">19:30:01.856</time>
Date and time
<time datetime="2016-01-20T19:30">7.30pm, 20 January 2016</time>
Date and time with timezone offset
<time datetime="2016-01-20T19:30+01:00">7.30pm, 20 January 2016 is 8.30pm in France</time>
Calling out a specific week number
<time datetime="2016-W04">The fourth week of 2016</time>

Semantic layout elements

<main>
Represents content unique to the page. It must be used only once per page and nested directly inside the <body>
<article>
Encloses a block of related content that makes sense on its own without the rest of the page
<section>
Similar to <article> but its more for grouping a single part of the page that constitutes one single piece of functionality (e.g. a mini map, or a set of article headlines and summaries). It's considered best practice to begin each section with a heading (<h1>, <h2>, <h3>...). You can place diferent sections inside articles or different articles inside sections
<aside>
Contents contet that isn't directly related to the main content but can provide additional information indirectly related to it (glossary, entries, author bio, related links etc)
<header>
Represents a group of introductory content. If it's a child of the body, it defines the global header of a webpage, if it's a child of an article or section it defines a specific header for that section.
<nav>
Contains the main navigation for the page, secondary links would not go here.
<footer>
Represents a group of end content for a page.

Tip: Consider a shopping cart widget that you could choose to pull up at any point. It isn't really an <aside> as it doesn't necessarily relate to the main content of the page. It doesn't even particularly warrant using a <section> as it isn't part of the main content of the page, so in this case we would use the non semantic <div> element.

<hr>

This creates a horizontal rule that denotes a thematic change in the text (such as a change on topic or scene)

<figure> and <figcaption>

These provides a semantic container for figures and clearly link figures to their captions. Captions and alt text shouldn't just say the same thing because they both appear when the image is gone.

Display Video Text Tracks

WebVTT is a format for writing text files containing multiple strings of text along with metadata such as what time in the video you want each text string to be displayed, and even limited styling/positioning information. These text strings are called cues, and you can display different types for different purposes, the most common being:

subtitles
Translations of foreign material, for people who don't understand the words spoken in the audio.
captions
Synchronized transcriptions of dialog or descriptions of significant sounds, to let people who can't hear the audio understand what is going on.
timed descriptions
Text for conversion into audio, to serve people with visual impairments.

Responsive images

There are two approaches for creating responsive images: Resolution switching and Art direction.

Resolution switching: Different sizes

With resolution switching (different sizes) we'll display identical image content, just larger or smaller depending on the device. We can use the srcset and sizes to provide several additional source images along with hints to help the browser pick the right one.

{% highlight html %} {% endhighlight %}
srcset
Defines the set of images we will allow the browser to choose between. For each image we write:
  1. An image filename (elva-fairy-320w.jpg)
  2. The image's inherent width in pixels (320w). This is the image's real size.
sizes
Defines a set of media conditions (screen widths) and indicates what image size would be best to choose, when certain media conditions are true. For each media condition we write:
  1. A media condition ((max-width: 320px)).
  2. The width of the slot the image will fill when the media condition is true (280px). For the slot width you may provide an absolute length (px, em) or a length relative to the viewport (vw) but not percentages. The last slot which has no media condition is the default that is chosen when none of the media conditions are true. Note: The browser ignores everything after the first matching condition.

So, with these attributes in place, the browser will:

  1. Look at its device width.
  2. Work out which media condition in the sizes list is the first one to be true.
  3. Look at the slot size given to that media query.
  4. Load the image referenced in the srcset list that most closely matches the chosen slot size.

Older browsers that don't support these features will just ignore them and load the image referenced in the src attribute.

Tip: In the <head> you should place <meta name="viewport" content="width=device-width">, this forces mobile browsers to adopt their real viewport width for loading web pages (some mobile browsers lie about their viewport width, and instead load pages at a larger viewport width, then shrink the loaded page down)

Resolution switching: Same size, different resolutions

If you're supporting multiple display resolutions, but everyone sees your image at the same real-world size on screen, you can allow the browser to choose an appropiate resolution image by using srcset with x-descriptions and without sizes:

{% highlight html %} {% endhighlight %}

In this case, sizes is not needed — the browser simply works out what resolution the display is that it is being shown on, and serves the most appropriate image referenced in the srcset. So if the device accessing the page has a standard/low resolution display, with one device pixel representing each CSS pixel, the elva-fairy-320w.jpg image will be loaded (the 1x is implied, so you don't need to include it.) If the device has a high resolution of two device pixels per CSS pixel or more, the elva-fairy-640w.jpg image will be loaded.

Art direction

With this approach we change the image displayed to suit different image display sizes. For example, if a large landscape shot with a person in the middle is shown on a website when viewed on a desktop browser, then shrunk down when the website is viewed on a mobile browser, it will look bad as the person will be really tiny and hard to see. It would probably be better to show a smaller, portrait image on mobile, which shows the person zoomed in.

For this, we use the <picture> element, containing several <source> elements that provide several different sources for the browser to choose between, followed by the all-important <img> element:

{% highlight html %} {% endhighlight %}

Note that <source> can take a srcset attribute with multiple images referenced, and a sizes attribute too. So you could offer multiple images via a <picture> element, but then also offer multiple resolutions of each one too. Realistically, you probably won't want to do this kind of thing very often.

Tip: You should use the media attribute only in art direction scenarios; when you do use media, don't also offer media conditions within the sizes attribute.

Form validation

HTML5 can validate most user data without relying on scripts, this is done by using validation attributes on form elements.

When an element is valid:

  • The element matches the :valid CSS pseudo-class.
  • The browser will submit the form, provided there is nothing else stopping it from doing so.

When an element is invalid:

  • The element matches the :invalid CSS pseudo-class.
  • The user won't send the form and will display an error message

Validation attributes:

  • required: Makes an input mandatory, the form won't submit if the input is empty.
  • pattern: Validates and input value against a Regular Expression
  • minlength and maxlength: Constrain in size <input> and <textarea> elements.
  • min and max: Provide validation constraint for number fields (inputs of type number).
  • step: This attribute is taken by the number and range types and specifies what increment the value will take when the input controls are used.

Tip: Some browsers don't support the number type so you could provide the \d+ pattern where \d means a digit, and + means 1 or more times so \d+ is one or more digits, in order to validate the input text as a number.

The browser will display error messages for invalid form widgets but there's no standard way to change their look and feel with CSS and they depend on browser locale, which means that you can have a page in one language but an error message displayed in another language.

The only way to customize the appearance and text of the messages is with JavaScript, and HTML5 provides the constraint validation API to check and customize the state of a form element.

lang

Helps define the language of an element: the language that non-editable elements are written in, or the language that the editable elements should be written in by the user.

A language tag is made of hyphen-separated language subtags, where each subtag indicates a certain property of the language. The 3 most common subtags are:

Language subtag
Required. A 2-or-3-character code that defines the basic language, typically written in all lowercase. For example, the language code for English is en, and the code for Badeshi is bdz.
Script subtag
Optional. This subtag defines the writing system used for the language, and is always 4 characters long, with the first letter capitalized. For example, French-in-Braille is fr-Brail and ja-Kana is Japanese written with the Katakana alphabet. If the language is written in a highly typical way, like English in the Latin alphabet, there is no need to use this subtag.
Region subtag
Optional. This subtag defines a dialect of the base language from a particular location, and is either 2 letters in ALLCAPS matching a country code, or 3 numbers matching a non-country area. For example, es-ES is for Spanish as spoken in Spain, and es-013 is Spanish as spoken in Central America. “International Spanish” would just be es.

To find the correct subtags visit:

  • https://r12a.github.io/app-subtags/
  • http://www.iana.org/assignments/language-subtag-registry/language-subtag-registry

For the CSS pseudo-class :lang, two invalid language names are different if their names are different. So while :lang(es) matches both lang="es-ES" and lang="es-419", :lang(xyzzy) would not match lang="xyzzy-Zorp!".

You should keep the tag as short as possible. For instance, use ja for Japanese and not ja-JP, unless there is a particular reason that you need to say that this is Japanese as spoken in Japan, rather than elsewhere.

Tip: You should always set the language of your page on the <html> element.

Detailed spec: w3 language tags

title

Contains text representing advisory information, related to the element it belongs to. Some typical uses:

  • Labeling <iframe> elements.
  • Providing an inline programmatically associated label for an <input> element.
  • Labeling controls in data tables.

If this attribute is omitted, it means that the title of the nearest ancestor of this element is still relevant (and could be used as the tooltip for that element). If this attribute is set to the empty string, it explicitly means its nearest ancestor's title is not relevant (and shouldn't be used in the tooltip for this element).

Additional semantics are attached to the title attributes of the following elements:

<link>
The title attribute is used to define a preferred or an alternate stylesheet.
Any stylesheet in a document falls into one of the following categories:
  • Persistent (no rel="alternate", no title=""): always applies to the document.
  • Preferred (no rel="alternate", with title="..." specified): applied by default, but disabled if an alternate stylesheet is selected. There can only be one preferred stylesheet, so providing stylesheets with different title attributes will cause some of them to be ignored.
  • Alternate (rel="stylesheet alternate", title="..." must be specified): disabled by default, can be selected.
<abbr>
The title attribute must contain a full human readable description or expansion of the abbreviation
Tip: The title attribute is not required in abbreviations, you can use <abbr> to semantically note an abbreviation without providing an expansion or description.
<input>
The title attribute is used in tandem with the pattern attribute, which is a regular expression that the control's value is checked against. The title attribute describes the pattern to help the user.

It's a good idea to add titles to links to provide supplementary useful information about the link, such as what kind of information the page contains, or things to be aware of.

Tip: Titles are revealed on mouse hover which are not shown while using keyboard controls so if the title info is important to the usability of the page you should put it in the regular text

Titles can also be added to images, however this does not come recommended, read The trials and tribulations of the title attribute

mailto <href> value

In its most basic and commonly used form, a mailto: link simply indicates the email address of the intended recipient. For example: <a href="mailto:nowhere@mozilla.org">Send email to nowhere</a>

In fact, the email address is even optional. If you leave it out (that is, your href is simply "mailto:"), a new outgoing email window will be opened by the user's mail client that has no destination address specified yet.

In addition to the email address, you can provide other information. In fact, any standard mail header fields can be added to the mailto URL you provide. The most commonly used of these are "subject", "cc", and "body" (which is not a true header field, but allows you to specify a short content message for the new email). Each field and its value is specified as a query term.

Here's an example that includes a cc, bcc, subject and body:

{% highlight html %} Send mail with cc, bcc, subject and body {% endhighlight %}

The values of each field must be URL-encoded, that is with non-printing characters (invisible characters like tabs, carriage returns, and page breaks) and spaces percent-escaped (Percent encoding). Also note the use of the question mark (?) to separate the main URL from the field values, and ampersands (&) to separate each field in the mailto: URL. This is standard URL query notation. Read The GET method (The GET method) to understand what URL query notation is more comonly used for. You can specify multiple addresses.

sandbox

Usable in embedded content like <iframe>, by using the sandbox attribute with no parameters you impose all available restrictions and if absolutely required, you can add permissions back one by one inside the sandbox attribute value. See the sandbox reference entry for all the available options. One important note is that you should never add both allow-scripts and allow-same-origin to your sandbox attribute — in that case the embedded content could bypass the same origin security policy that stops sites from executing scripts, and use JavaScript to turn off sandboxing altogether.

Note that you should Always use the sandbox attribute.

Also read: On the X-Frame-Options Security Header

enctype

This lets you specify the value of the Content-Type HTTP header included in the request generated when the form is submitted. By default, its value is application/x-www-form-urlencoded. In human terms, this means: "This is form data that has been encoded into URL parameters."

In order to send files, which are binary data instead of text data you must:

  • Set the method attribute to POST because file content can't be put inside URL parameters.
  • Set the value of enctype to multipart/form-data because the data will be split into multiple parts, one for each file plus one for the text data included in the form body (if text is also entered into the form).
  • Include one or more File picker widgets to allow your users to select the file(s) that will be uploaded.

Tips

  • Links can wrap block level elements. Example:

  • You can quite happily have two index.html files in one project as long as they are in different locations in the filesystem. Many web sites do. The second index.html would perhaps be the main landing page for project-related information.

  • In order to improve speed, it's a good idea to set the iframe's src attribute with JavaScript after the main content is done loading. This makes your page usable sooner and decreases your official page load time (an important SEO metric.)
